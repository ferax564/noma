# Noma v0.7.0 — §24.9 Papercuts Bundle Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the four ⏳ items tracked in PLAN.md §24.9 by shipping a single v0.7.0 release: shared-asset site renderer, trusted-publishing manifest gate, `noma diff` command emitting `::state_change` blocks for attribute drift, and marketplace-publish prep for `noma-language`. (The actual marketplace publish itself is an interactive post-release step the user runs — Task 5 keeps the release notes honest about that.)

**Architecture:** Each item is an additive change inside an existing module — no AST variants added. (1) `renderer-html.ts` gains an optional `stylesheetHref` so callers can opt into `<link>` instead of inlined `<style>`; `renderer-site.ts` uses it to write `_assets/theme.css` once. (2) `book.ts` exposes a `trustedPublishing` flag the CLI honors as an implicit `--no-unsafe`; the manifest is the final word — there is no CLI flag to re-enable escape hatches once the manifest forbids them. (3) A new `src/diff.ts` walks two AST snapshots and emits `::state_change` directives **only for attribute-value drift on blocks identified by `id` and present in both snapshots**. Block adds, deletes, renames, prose changes, and ID-less directives are explicitly out of scope for v0.7 (tracked for v0.7.1+). The diff CLI requires an explicit `--at YYYY-MM-DD` so output is deterministic. (4) Marketplace-publish prep: `vsce` packaging metadata, README polish, version bump, locally-built `.vsix` verification. Task 5 emits release notes that say "publish prep" until the user confirms the live marketplace publish completed.

**Tech Stack:** TypeScript 5.7 (`tsx` runner, `tsc --noEmit`), `node:test`, `js-yaml`, Puppeteer for PDF, VS Code `vsce` (added as a tool-folder devDep).

## Task 1: Trusted-publishing manifest field (smallest, foundational)

**Files:**
- Modify: `src/book.ts:9-18` (extend doc-comment on `BookManifest`; no typed field — see rationale below)
- Modify: `src/cli.ts:227-246` (honor flag on `--to site`)
- Modify: `src/cli.ts:248-272` (honor flag on single-page `render --to html` when input is a manifest)
- Create: `test/book-trusted-publishing.test.ts`
- Modify: `examples/book/book.noma.yml` (leave alone — keep default behavior)

**Rationale for no typed `trustedPublishing` field:** the YAML key is `trusted_publishing` (snake_case is the manifest idiom; the doc convention matches what users already write). `loadBookChapters` returns the raw manifest as `Record<string, unknown>`, and the CLI reads `manifest.trusted_publishing` directly. Adding a typed camelCase `trustedPublishing?: boolean` to `BookManifest` would document a property no caller reads — better to document the YAML key in the existing interface JSDoc and leave the runtime read at the manifest-key boundary.

**Rationale for no CLI override flag:** the manifest is the final word on trust. The plan originally said "explicit CLI flags still take precedence" but the only existing CLI flag (`--no-unsafe`) is one-way (it can only tighten). Adding a `--unsafe-allow` flag to override the manifest would defeat the security posture the manifest exists to enforce. If a user needs to render the same source without the trust restriction, they edit the manifest or render the chapter directly without going through the manifest.

The chapter pages currently come back from `renderHtml(...)` with a `<style>${themeCss}</style>` baked in (`renderer-html.ts:198`). Instead of regex-replacing that string in the site renderer (brittle — breaks if `renderHtml`'s output format ever shifts), extend `renderHtml` with a `stylesheetHref` option. When set, `renderHtml` emits `<link rel="stylesheet" href="...">` and ignores `themeCss`. The site renderer then drives this cleanly and the single-page path is unchanged.

**Slug-with-slash note:** chapter slugs in book mode can contain `/` because of scoped IDs (`risk-premia-3/risks`). For the `--to site` output, only the chapter-root slug (level-1 section / filename) is used as the page filename — scoped IDs are anchor fragments, not file paths. So `_assets/theme.css` is always a valid relative URL from `${slug}.html` (single directory deep). If a future change lets chapter pages nest into subdirs, switch to a per-page relative path. For v0.7, document the assumption.

export function renderSite(
  manifest: Record<string, unknown>,
  chapters: LoadedChapter[],
  outDir: string,
  options: SiteRenderOptions = {},
): void {
  const absOut = resolve(outDir);
  mkdirSync(absOut, { recursive: true });

  const themeCss = options.themeCss ?? "";
  const hasTheme = themeCss.length > 0;
  if (hasTheme) {
    const assetsDir = join(absOut, "_assets");
    mkdirSync(assetsDir, { recursive: true });
    writeFileSync(join(assetsDir, "theme.css"), themeCss, "utf8");
  }

  const idMap = buildIdMap(chapters);
  const bookTitle =
    options.title ||
    (typeof manifest.title === "string" ? manifest.title : undefined) ||
    "Noma Book";

  for (const ch of chapters) {
    const html = renderHtml(ch.doc, {
      standalone: true,
      ...(hasTheme ? { stylesheetHref: THEME_HREF } : { themeCss: "" }),
      title: chapterTitle(ch) || bookTitle,
      allowEscapeHatches: options.allowEscapeHatches !== false,
      ...(options.math ? { math: options.math } : {}),
    });
    const rewritten = rewriteWikilinks(html, ch.slug, idMap);
    const withNav = injectNav(rewritten, chapters, ch.slug, bookTitle);
    const target = join(absOut, `${ch.slug}.html`);
    mkdirSync(dirname(target), { recursive: true });
    writeFileSync(target, withNav, "utf8");
  }

  const indexHtml = renderIndex(bookTitle, manifest, chapters, hasTheme, idMap);
  writeFileSync(join(absOut, "index.html"), indexHtml, "utf8");
}

  manifest: Record<string, unknown>,
  chapters: LoadedChapter[],
  hasTheme: boolean,
  idMap: Map<string, IdLocation>,

  const author =
    typeof manifest.author === "string" ? manifest.author : undefined;
  const nav = buildNav(chapters, null, bookTitle);
  const items = chapters
    .map((c) => {
      const label = chapterTitle(c) ?? c.slug;

  ${author ? `<p class="noma-site-author">${escapeHtml(author)}</p>` : ""}
</header>
<ol class="noma-site-toc">
${items}
</ol>
</main>
</body>
</html>`;

## Task 3a: `noma diff` — module + smallest test (attr change)

**Files:**
- Create: `src/diff.ts`
- Create: `test/diff.test.ts`

The diff algorithm is structural: walk both docs by ID, compare directive nodes, emit `state_change` for each scalar attribute whose value changed **on both sides**. Attribute add (only-after) and attribute delete (only-before) are out of scope for v0.7 — `state_change` validity requires both `from=` and `to=` per the validator at `src/validator.ts:237`, and there is no agreed-upon sentinel for "the attribute didn't exist." Tracked for v0.7.1: pick a sentinel (likely `from="(unset)"` / `to="(unset)"`), document it in the spec, and lift the skip. Other v0.7 non-goals: prose changes, heading-text changes, block adds/deletes, block renames, ID-less directives, and nested-children diffs.

**Determinism note.** `diffDocs` does not synthesize `at=` automatically. The CLI requires `--at YYYY-MM-DD`; the programmatic API requires `options.at`. Same inputs always produce the same output bytes.

**Duplicate-id behavior.** If a document contains two directives with the same `id=`, the diff throws — the validator already flags this as an error and the diff can't reason about which copy to compare. Document this in the JSDoc.

Create `src/diff.ts`:

```ts
import type { Attrs, AttrValue, DirectiveNode, DocumentNode } from "./ast.js";
import { isDirective, walk } from "./ast.js";

export interface DiffOptions {
  /**
   * Required timestamp written as `at=` on every emitted state_change. Format:
   * `YYYY-MM-DD` (or any ISO-8601 date string). Required so output is
   * deterministic — same inputs always produce the same bytes.
   */
  at: string;
  /**
   * Optional reason string applied to every emitted state_change. Surfaces in
   * the rendered narrative; agents typically set this to a commit message or
   * a refresh trigger description.
   */
  reason?: string;
}

/**
 * Compare two parsed documents and emit a flat list of synthesized
 * `::state_change` directives describing every scalar attribute drift on
 * blocks identified by `id` and present in both snapshots.
 *
 * v0.7 scope: only attributes defined on both sides where the value differs.
 * Out of scope: attribute add (only-after), attribute delete (only-before),
 * prose/heading changes, block adds/deletes, block renames, ID-less
 * directives, nested-children diffs. Tracked for v0.7.1.
 *
 * Throws if either document contains duplicate IDs (the validator already
 * flags this as an error; diff cannot reason about which copy to compare).
 */
export function diffDocs(
  before: DocumentNode,
  after: DocumentNode,
  options: DiffOptions,
): DirectiveNode[] {
  if (!options || typeof options.at !== "string" || options.at.length === 0) {
    throw new Error("diffDocs: options.at is required (YYYY-MM-DD)");
  }
  const at = options.at;
  const beforeIdx = indexById(before);
  const afterIdx = indexById(after);
  const deltas: DirectiveNode[] = [];

  for (const [id, afterNode] of afterIdx) {
    const beforeNode = beforeIdx.get(id);
    if (!beforeNode) continue;
    for (const delta of diffAttrs(beforeNode.attrs, afterNode.attrs)) {
      deltas.push(makeStateChange(id, delta, at, options.reason));
    }
  }

  return deltas;
}

interface AttrDelta {
  attribute: string;
  from: AttrValue;
  to: AttrValue;
}

function diffAttrs(a: Attrs, b: Attrs): AttrDelta[] {
  const keys = new Set([...Object.keys(a), ...Object.keys(b)]);
  keys.delete("id");
  const out: AttrDelta[] = [];
  for (const k of [...keys].sort()) {
    const av = a[k];
    const bv = b[k];
    // v0.7: only emit deltas when the attribute exists on BOTH sides.
    // state_change validation requires both `from=` and `to=` (validator.ts:237).
    if (av === undefined || bv === undefined) continue;
    if (av === bv) continue;
    out.push({ attribute: k, from: av, to: bv });
  }
  return out;
}

function indexById(doc: DocumentNode): Map<string, DirectiveNode> {
  const map = new Map<string, DirectiveNode>();
  for (const node of walk(doc)) {
    if (!isDirective(node) || !node.id) continue;
    if (map.has(node.id)) {
      throw new Error(`diffDocs: duplicate id "${node.id}" — fix with \`noma check\` first`);
    }
    map.set(node.id, node);
  }
  return map;
}

function makeStateChange(
  blockId: string,
  delta: AttrDelta,
  at: string,
  reason: string | undefined,
): DirectiveNode {
  const attrs: Attrs = {
    block: blockId,
    attribute: delta.attribute,
    from: delta.from,
    to: delta.to,
    at,
  };
  if (reason) attrs.reason = reason;
  return {
    type: "directive",
    name: "state_change",
    attrs,
    children: [],
  };
}
```

Syntax highlighting and language configuration for the [Noma](https://ferax564.github.io/noma/) document format (`.noma`).

## Features

- Directive blocks (`::name{...}` / `::`) with attribute highlighting for the patch-protocol-aware attrs (`id=`, `for=`, `parent=`, `block=`, `dataset=`, `column=`, `src=`, `href=`, `aliases=`).
- Headings with optional `{id="..." aliases="a,b"}` suffix.
- Wikilinks `[[block-id]]`.
- Math: `$..$`, `$$..$$`, `\(..\)`, `\[..\]`, and `::math` (embedded LaTeX).
- Pipe tables, list items, block quotes, fenced code.
- Embedded language scopes:
  - YAML frontmatter
  - JSON inside `::plotly`
  - LaTeX inside `::math`
  - Mermaid inside `::diagram{kind="mermaid"}`
  - DOT inside `::diagram{kind="graphviz"}`
- Escape-hatch directives (`::html`, `::svg`, `::script`) marked with `invalid.illegal.*` scopes so themes can warn on them.
- Folding markers track directive opener/closer pairs.

## Install

From the VS Code marketplace:

```
ext install ferax564.noma-language
```

Or from a local `.vsix`:

```bash
cd tools/vscode-noma
npx vsce package
code --install-extension noma-language-0.2.0.vsix
```

## Companion CLI

The [`@noma/cli`](https://github.com/ferax564/noma) package provides `noma parse|render|check|patch|diff|verify` on the same `.noma` sources this extension highlights.
